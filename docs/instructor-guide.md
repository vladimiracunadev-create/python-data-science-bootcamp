# Guia del instructor

> Audiencia: docente principal, docente apoyo, coordinacion academica.

## Objetivo

Traducir la metodologia del bootcamp a decisiones concretas antes, durante y despues de la clase.

## Checklist previo a cada sesion

| Area | Que revisar |
|---|---|
| Contenido | objetivo de la sesion, notebook, ejercicios, variaciones |
| Entorno | app local, datasets, runner y notebooks base |
| Aula | tiempos, agrupamiento, equipamiento y conectividad |
| Riesgos | errores probables, dudas frecuentes y apoyo para quienes se bloquean |

## Antes de la clase

1. revisar el objetivo visible de la sesion;
2. probar el notebook y la ruta critica del ejemplo;
3. confirmar que el dataset correcto esta disponible;
4. decidir cual sera el minimo comun del grupo;
5. preparar un desafio opcional para quienes avancen mas rapido.

## Playbook de clase

### Apertura

- conectar con la sesion anterior;
- explicar para que sirve lo que se vera hoy;
- mostrar el resultado esperado;
- bajar la ansiedad con una tarea inicial acotada.

### Desarrollo

- usar un ejemplo breve antes de abrir variaciones;
- verbalizar decisiones mientras escribes;
- preguntar "que creen que pasara" antes de ejecutar;
- evitar meter mas de una idea nueva fuerte por bloque.

### Practica

- pasar rapidamente por los puestos o por parejas;
- detectar si el problema es tecnico o de comprension;
- no resolverle todo a quien se bloquea: dar el siguiente paso, no la solucion completa;
- recoger un error comun para revisarlo en plenario.

### Cierre

- pedir una conclusion, no solo "me funciono";
- marcar un error frecuente y como leerlo;
- dejar claro que se espera para la siguiente sesion.

## Matriz de intervencion docente

| Situacion | Senal visible | Intervencion sugerida |
|---|---|---|
| Miedo a equivocarse | silencio, espera pasiva, no ejecuta | victoria rapida, pregunta de baja exposicion |
| Copia sin comprension | resultado correcto pero no sabe explicarlo | pedir adaptacion y verbalizacion |
| Ritmos muy distintos | parte del grupo termina muy rapido | minimo comun + desafio opcional |
| Frustracion por error | abandono o repeticion mecanica | leer el error en voz alta y aislar la causa |
| Dispersion por tecnologia | cambia de herramienta, busca demasiado | volver al objetivo y acotar el alcance |

## Buenas practicas para explicacion tecnica

- hablar en voz alta mientras escribes;
- comparar lo correcto con lo casi-correcto;
- usar nombres de variables que expliquen intencion;
- no ocultar errores de sintaxis comunes;
- mostrar al menos una interpretacion del resultado obtenido.

## Integracion con evaluacion

El docente debe recoger evidencia de:

- ejecucion minima lograda;
- comprension del procedimiento;
- interpretacion del resultado;
- autonomia frente a pequenas variaciones.

## Despues de la clase

1. registrar que se logro y que quedo fragil;
2. anotar errores recurrentes;
3. decidir si la siguiente sesion mantiene ritmo o necesita ajuste;
4. guardar notebooks o evidencia utiles para la siguiente cohort.

## Regla de oro

Una buena clase no es la que "cubre mas contenido". Es la que deja al grupo entendiendo una idea util, pudiendo hacer algo con ella y con menos miedo a seguir aprendiendo.
