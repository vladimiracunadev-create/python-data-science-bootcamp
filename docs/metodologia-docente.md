# 🧠 Metodologia docente

> Audiencia: docente, coordinacion academica, evaluador pedagogico.

## Proposito

Este documento define como se ensena dentro de este bootcamp. No describe solo contenidos, sino la forma de mediarlos para que el estudiante avance con comprension, practica y confianza.

## Idea fuerza

El bootcamp no se apoya en "mostrar codigo" como fin. Su foco es convertir herramientas tecnicas en aprendizaje real mediante:

- progresion controlada;
- contexto comprensible;
- practica frecuente;
- interpretacion de resultados;
- retroalimentacion que ayude a avanzar.

## Principios metodologicos

### 1. De la comprension minima al uso significativo

Cada sesion debe dejar al estudiante pudiendo hacer algo util y explicarlo. La dificultad aumenta cuando ya existe una base minima de seguridad.

### 2. Contexto antes que tecnicismo

Los conceptos se presentan ligados a problemas entendibles: ventas, asistencia, transporte, soporte o seguimiento de estudiantes. Eso reduce friccion y hace visible para que sirve lo que se aprende.

### 3. Mostrar, practicar, interpretar

La secuencia didactica central del programa es:

1. presentar un objetivo claro;
2. mostrar un ejemplo corto;
3. practicar con acompanamiento;
4. pedir interpretacion, no solo ejecucion.

### 4. Error como material de trabajo

Los errores no se esconden. Se usan para ensenar lectura de mensajes, depuracion, comparacion de alternativas y autonomia gradual.

### 5. Un minimo comun para todos

La clase debe asegurar un piso de logro compartido, aun cuando haya estudiantes que avancen mas rapido que otros.

### 6. Tecnologia como apoyo, no como reemplazo

El programa acepta buscadores, asistentes o herramientas de IA, pero siempre subordinadas al objetivo de aprendizaje. La regla es:

1. pensar;
2. consultar;
3. verificar;
4. explicar.

## Modelo de sesion de 90 minutos

| Momento | Tiempo estimado | Que ocurre |
|---|---|---|
| Apertura | 10 min | objetivo visible, activacion breve y baja de ansiedad |
| Explicacion guiada | 15 a 20 min | una idea fuerte, con lenguaje simple y ejemplo claro |
| Demostracion | 15 a 20 min | el docente modela el proceso y verbaliza decisiones |
| Practica acompanada | 20 a 25 min | el grupo replica, modifica y pregunta |
| Chequeo de comprension | 10 a 15 min | variacion corta, ticket de salida o explicacion oral |
| Cierre | 5 min | sintesis, error comun y siguiente paso |

## Rol del docente

El docente cumple al mismo tiempo cinco funciones:

1. traduce contenido tecnico a ejemplos comprensibles;
2. detecta bloqueos de comprension y no solo errores de sintaxis;
3. regula el ritmo para no perder ni al grupo lento ni al grupo rapido;
4. normaliza la equivocacion como parte del proceso;
5. protege el foco pedagogico cuando la tecnologia distrae o sobrecomplica.

## Tipos de evidencia que la clase debe producir

| Tipo de evidencia | Ejemplo |
|---|---|
| Evidencia de ejecucion | notebook, script, grafico o tabla |
| Evidencia de comprension | explicar una linea, un filtro o un resultado |
| Evidencia de interpretacion | escribir un hallazgo en lenguaje simple |
| Evidencia de transferencia | resolver una pequena variacion del ejercicio |

## Intervenciones pedagogicas clave

### Cuando el grupo va lento

- reducir cantidad de variaciones;
- reforzar una sola idea por bloque;
- usar ejemplos ya preparados;
- convertir el objetivo en una tarea mas acotada.

### Cuando el grupo va rapido

- agregar desafio opcional;
- pedir comparacion entre dos soluciones;
- pedir interpretacion adicional o justificacion;
- abrir una micro extension del problema.

### Cuando aparece ansiedad

- mostrar primero una victoria rapida;
- hacer visible que nadie debe dominar todo de inmediato;
- preguntar con baja exposicion;
- trabajar en parejas antes de pasar a plenario.

### Cuando la tecnologia reemplaza el razonamiento

- volver al objetivo de la tarea;
- pedir hipotesis antes de consultar;
- pedir adaptacion y explicacion de cualquier respuesta obtenida;
- bajar la complejidad al nivel del grupo.

## Errores frecuentes que se deben anticipar

| Tipo de error | Ejemplos | Respuesta docente recomendada |
|---|---|---|
| Sintaxis | parentesis, comillas, `=` vs `==` | leer el error juntos y mostrar patron |
| Modelo mental | no entender que hace un filtro o un `groupby` | volver al dato visible y al objetivo |
| Interpretacion | creer que el grafico "habla solo" | pedir una conclusion escrita u oral |
| Dependencia externa | copiar codigo sin criterio | pedir ajuste, justificacion y explicacion |

## Indicadores de exito de una buena sesion

- el estudiante entiende que hizo y para que sirve;
- puede modificar algo pequeno sin perderse;
- puede leer el resultado y decir que significa;
- gana confianza para seguir practicando;
- no depende exclusivamente de copiar pasos.
